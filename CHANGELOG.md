- **fix(skill): add Greptile-service-errored recovery procedure to deft-directive-swarm Phase 6** (#526): Added explicit detection + retry-once + three-way escalation procedure for the Greptile-service-errored state to `skills/deft-directive-swarm/SKILL.md` Phase 6 Step 1. New `!` rule detects the literal "Greptile encountered an error while reviewing this PR" comment body on the current HEAD, warns that the GitHub CheckRun COMPLETED/NEUTRAL is NOT a passing signal in this case, requires a single `@greptileai review` retry with a 10-minute cap, and on a second error escalates to the user with a three-way choice: (a) wait longer ~15-20 min, (b) push an empty `chore: retrigger greptile` commit to force a fresh pass, (c) merge with a documented override where the rationale MUST be recorded in the merge commit body (not just the PR body) citing prior Greptile success on a pre-rebase SHA + CI/Go + CI/Python success on the current SHA + pure conflict-resolution rebase with no new business logic. Extended the existing Phase 6 Step 1 Greptile gate wording so "a stale or in-progress review is not sufficient; an errored review is also not sufficient; follow the escalation procedure" is explicit. Added a polling-sub-agent contract requiring short-lived Phase 6 pollers to detect the error sentinel and emit a distinct `PR #<N> Greptile errored` message to the parent monitor, and to separately track "last-reviewed SHA" vs "errored on current HEAD" so an errored state on HEAD is not masked by a successful review on a prior SHA. Updated Phase 6 Step 5 Slack release announcement template and populate rules so any merge that used the override path is explicitly called out on a new `*Override merges*` line with the one-line rationale from the merge commit body (omitted only when no PR in the release used the override path), making the documented override trail traceable from release notes. Added four anti-patterns: treating COMPLETED/NEUTRAL as a passing review, indefinite errored-state looping / silent poll-cap timeout, merging on the NEUTRAL CheckRun alone when the comment body is the error sentinel, and omitting override-merged PRs from the Slack announcement. New content tests in `tests/content/test_skills.py` assert the detection rule, retry-once + 10-minute cap, three-way escalation, gate-wording extension, explicit `errored` monitor exit, NEUTRAL-alone merge prohibition, polling sub-agent contract, NEUTRAL-CheckRun anti-pattern, and the Slack override-merge callout are all present.
